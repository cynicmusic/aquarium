## Diagnosis (already done — context for future me)

The cuttlefish-preview render loop lives at [src/workshop/CuttlefishPreview.js:778](../../src/workshop/CuttlefishPreview.js#L778). It runs ONE renderer, ONE RAF. The main aquarium scene at [src/core/Scene.js:296](../../src/core/Scene.js#L296) uses `setAnimationLoop`, also single. Toggling/removing the params panel does NOT spin up a second render loop — that was a hypothesis we ruled out.

The chromatophore fragment shader [src/shaders/ChromatophoreMaterial.js:34](../../src/shaders/ChromatophoreMaterial.js#L34) is the dominant cost. Per fragment, every frame, it always runs:

- 5-octave FBM (called from `expansionWave`, `iridophoreLayer`, `forkLocal`, `headMask`, `stripeMottle`)
- Two voronoi calls (`chromatophoreLayer` warm + `chromatophoreLayer2` cool), each scanning 25 cells with 2 hash calls per cell
- `forkLocal` zebra (3-octave bar noise + 3-octave fbm)
- `headMask` reticulation (5+4 octaves) — only short-circuits AFTER the gate test, but most of fbm has already run
- `stripeMottle` (4+3 octaves)
- iridophore spectral
- sparkle layer (cell-grid + flicker)

Even when a uniform like `uChromaIntensity` is 0 (e.g. on the side fins), the voronoi+FBM still runs. **GLSL compilers can't constant-fold uniforms** — that's why the wins below are about `#define`-stripping at material-create time, not `if (uniform == 0)` checks.

The cuttlefish builds **5 separate ShaderMaterial instances** ([Cuttlefish.js:848, 880, 897, 987, 1018](../../src/entities/Cuttlefish.js#L848)) with different uniform sets but the same shader source. Three.js shares the compiled program across them, so this isn't the issue — but it does mean each has its own draw call, and each runs the full shader unless we vary the source/`defines` per role.

SwimWorld spawns 240 plants + 180 coral + 28 fish. Plants and coral already share geometry+material per preset ([swim/coral.js:138, swim/plants.js:375](../../src/swim/coral.js#L138)) but they're 420 SEPARATE Mesh objects = 420 draw calls. Instancing collapses that to ~20.

Pixel ratio at [Scene.js:36](../../src/core/Scene.js#L36) and [CuttlefishPreview.js:103](../../src/workshop/CuttlefishPreview.js#L103) is `Math.min(window.devicePixelRatio, 2)`. iPhone Pro Max has DPR=3 → clamped to 2 → 4× pixel area vs. DPR=1. With a fragment-heavy shader that's ~4× the cost.

### 1. `#define`-strip chromatophore shader for tentacles / arms / fins

**Why:** Tentacles, arms, and side fins all use the chromatophore shader with `zebraIntensity:0` and (for fins) `chromaIntensity:0`. But the GLSL still computes the full fork-local zebra, head reticulation, voronoi×2, sparkle, etc. on every fragment. Adding compile-time `#define` flags lets us drop entire functions on the surfaces that don't need them.

**Files:**
- [src/shaders/ChromatophoreMaterial.js](../../src/shaders/ChromatophoreMaterial.js) — wrap zebra/head/voronoi/sparkle blocks in `#ifdef HAS_*`, accept a `defines` opts param.
- [src/entities/Cuttlefish.js](../../src/entities/Cuttlefish.js) — at each createChromatophoreMaterial site, pass a `defines` set matching what the surface actually uses.

**Plan:**
- Add defines to the shader: `HAS_ZEBRA`, `HAS_HEAD_MASK`, `HAS_CHROMA`, `HAS_IRIDO`, `HAS_SPARKLE`. (FBM/voronoi helpers stay always-on since multiple features share them.)
- Wrap the relevant `void main()` sections + the function definitions of `zebraMask`, `headMask`, `chromatophoreLayer`, `chromatophoreLayer2`, `iridophoreLayer`, `sparkleLayer` in `#ifdef`.
- `createChromatophoreMaterial(opts)` accepts opts.defines OR auto-derives them from the intensity opts being non-zero.
- Mantle: all defines on. Head: no zebra. Fins: no zebra, no chroma. Tentacles + arms: no zebra, no head_mask. Tune as we go.

**Expected impact:** Big — this is the single biggest win on mobile. The 8 arms + 2 tentacles + 2 fins can drop to a tiny shader.

**Test:**

- Then deploy to staging via `./scripts/publish-cuttlefish.sh` and load on iPhone.
- Use the FPS counter (or add one) to measure before/after.

**Rollback:** revert the chromatophore + cuttlefish files. Defines are additive, no data loss risk.

Collapsing each preset into a single InstancedMesh drops that to ~20.

### 3. Cap WebGL backing-store pixel ratio at 1 on touch devices

**Why:** iPhone Pro Max DPR is 3 (clamped to 2 today = 4× pixel cost). Dropping the WebGL backing-store DPR to 1 cuts fragment cost ~4× without changing the canvas's CSS size — so the viewport / framing / aspect ratio stays identical, just slightly softer. (User explicitly said: don't change the viewport.)

**Files:**
- [src/core/Scene.js:36](../../src/core/Scene.js#L36) — `this.renderer.setPixelRatio(...)`
- [src/workshop/CuttlefishPreview.js:103](../../src/workshop/CuttlefishPreview.js#L103) — `renderer.setPixelRatio(...)`

**Change:**
```js
const isTouch = matchMedia('(pointer: coarse)').matches;
const maxDPR  = isTouch ? 1 : 2;
renderer.setPixelRatio(Math.min(window.devicePixelRatio, maxDPR));
```

**Why this preserves viewport:** `setPixelRatio` only affects the size of the WebGL framebuffer — it does NOT call `setSize`, so the `<canvas>` element's CSS width/height / position / aspect are untouched. Browser scales the smaller framebuffer up to canvas size with bilinear filtering. Visually = slightly softer, geometrically = identical.

**Expected impact:** ~3–4× fragment cost reduction on mobile. Should move iPhone from 1–3 fps into double digits. Zero change on desktop.

**Test:** On iPhone, before/after FPS. Visual check — accept that it's a hair softer; if it's intolerable we lift `maxDPR` to 1.5.

**Rollback:** revert the two lines. Trivial.

**Status:** 🧪 AWAITING USER TEST (implemented 2026-04-26)

**What landed:**
- [src/core/Scene.js](../../src/core/Scene.js) and [src/workshop/CuttlefishPreview.js](../../src/workshop/CuttlefishPreview.js) now detect coarse-pointer/touch devices and cap WebGL backing-store DPR at 1. Non-touch devices still cap at 2.
- This only changes `renderer.setPixelRatio(...)`; `setSize(...)`, camera, canvas CSS size, and viewport framing are untouched.

### 4. Stop recomputing fin vertex normals every frame

**Why:** [Cuttlefish.js:1119](../../src/entities/Cuttlefish.js#L1119) calls `geo.computeVertexNormals()` every frame on each fin. The fin has `(80+1)*(6+1)=567` verts → ~1130 triangles. computeVertexNormals iterates all triangles, accumulates normals on shared verts, and re-normalizes. JS-side cost on a small phone CPU adds up over 60 fps.

The fin wave amplitude is ≤0.3 at the outer edge and tiny at the root. The original normals (from buildSideFin) are good enough — the wave doesn't tilt the surface enough to matter for shading. Drop the per-frame recompute; if shading looks too flat, do it every Nth frame.
